1. System architecture

The three-tier view. Frontends on the left, backend services in the middle, integrations on the right. The new Asset Accounting work introduces new services (Rules Engine, Repair Tracker) and new outbound integrations (ERP chargeback) but doesn't change the fundamental shape of the system.

What's new in this picture

  • Rules engine — a new service that watches LossEvent inserts in Postgres, evaluates the active ThresholdRules, and emits actions (block flag, email, webhook). Stateless; can be deployed as a background worker.
  • Repair tracker — a new service that owns the repair queue, vendor assignments, and ETAs. Talks to ITSM if integration is on.
  • ERP integration — new outbound webhook for chargeback ledger entries. Batched daily by default.
  • Photo storage — S3 for the optional photo-at-return feature.
Design choice to flag for engineering: the Rules Engine should re-evaluate on LossEvent insert only, not on every event type. Otherwise it becomes a hot path on every checkout. A Postgres LISTEN/NOTIFY on the loss_events table is the cleanest trigger.

Existing tables that get new columns

  • device_models — add replacement_cost_cents, salvage_cents, expected_lifespan_months.
  • users — derived column or materialized view lifetime_loss_cents for fast lookup; updated by trigger on loss_events.
  • departments — derived quarter_loss_cents by trigger.
Pricing snapshot is critical. Store cost_basis_snapshot as JSONB on the LossEvent row itself. That snapshot is what gets reported on, not the current model price. If the iPhone 16e price changes from $599 → $649 next year, old loss reports still show the historical $599 because the snapshot is frozen at event time.

3. Return / damage event sequence

The path a damaged-return event takes through the system. This is the flow your devs will reference most when building.

sequenceDiagram autonumber participant U as User participant K as Kiosk participant API as API participant DB as Postgres participant RE as Rules Engine participant N as Notifications participant E as ERP participant R as Repair Tracker U->>K: Tap badge to return device K->>API: GET /orders/active?user_id=X API->>DB: SELECT open order DB-->>API: Order #N API-->>K: Order details K->>U: "Is the device damaged?" U->>K: Yes K->>API: GET /device_models/{id}/components API-->>K: [screen $179, port $89, battery $65, ...] K->>U: Show component picker U->>K: Select screen + port K->>API: POST /loss_events {order_id, components:[s,p], type:"damaged_repairable"} API->>DB: INSERT loss_events (snapshot cost basis) API->>DB: INSERT damage_components (2 rows) API->>DB: INSERT chargeback_ledger entry API->>DB: INSERT repair_tickets (stage: queued) API->>DB: NOTIFY loss_event_inserted Note over DB,RE: Rules engine wakes on NOTIFY RE->>DB: SELECT user lifetime, dept quarter RE->>DB: SELECT active threshold_rules RE->>RE: Evaluate triggers alt Threshold crossed RE->>DB: INSERT rule_fire_log par RE->>N: Send email + Slack N-->>U: Notification and RE->>API: Set user.checkout_blocked = true end end par Repair flow R->>R: Pick up new queue row R-->>API: Ticket created and Chargeback flow E->>E: Daily batch E->>API: Confirm posting end API-->>K: 200 OK K->>U: "Total $268 charged to Troy ER · Device sent to repair"

Where things can go wrong

  • Step 4 (component lookup) — if the component catalog is missing for a device model, default to the "full loss" choice instead of an empty picker. Don't let the operator get stuck.
  • Step 11 (NOTIFY → Rules Engine) — handle the case where the engine is down. Use a poll-fallback every 30s on a "pending" flag so events aren't lost.
  • Step 18 (ERP webhook) — webhook delivery should be retried with exponential backoff. Failed webhooks visible in an admin error queue, not silently dropped.
  • Steps 13-17 (parallel fan-out) — each side effect is independent; a failure in Slack delivery must not block the chargeback ledger insert or repair queue creation. Use a transactional outbox pattern.

The five exits from a LossEvent (in order of priority)

  1. Loss & damage reports. Every report query reads loss_events directly. No projection layer in MVP — keep it simple.
  2. Chargeback ledger. Insert happens in the same transaction as the LossEvent insert. Outbound webhook is async (transactional outbox).
  3. Repair queue. Only fires if disposition = repairable. Creates a ticket row with stage = queued.
  4. User & department running totals. Trigger-maintained columns. Fast read on every checkout to check thresholds.
  5. Rules engine re-evaluation. Async — triggered by NOTIFY, runs in milliseconds. Can fan out to blocks, warnings, notifications.

Recovery (the reverse path)

When a lost device is recovered, the recovery_event creates a reverse-direction LossEvent record with a negative amount. Chargeback ledger gets a credit entry. User's lifetime total goes down. Same data model, just signed amounts.

5. Glossary — new entities and concepts

TermMeans
Device ModelThe make/type record. iPhone 16e, AiRista badge, Pixel 8 Pro. Carries replacement cost, salvage, lifespan, and the repair component catalog.
Repair ComponentA subordinate row on a Device Model. Has a name and a cost. Example: {model: iPhone 16e, name: "Screen replacement", cost: $179}.
Damage ComponentThe link table between a LossEvent and Repair Component(s). Tells you which specific components were damaged in this incident.
LossEventThe accounting fact. Created when something happens that costs the customer money: damage at return, lost device, stolen device. Carries the cost-at-time-of-event snapshot.
Cost basis snapshotA frozen copy of the Device Model and Repair Component prices as they existed when the LossEvent was created. Reports always read from the snapshot, never from current prices. Prevents reports from changing retroactively.
Chargeback LedgerA separate accounting ledger that mirrors LossEvent. Outbound to ERP. Decoupled so finance can hold/reverse entries without modifying the source LossEvent.
Repair TicketWorkflow row that exists only when the LossEvent's disposition is repairable. Tracks vendor, stage, ETA.
Threshold RuleUser-configured "when X, do Y" rule. Scope is user / department / device. Trigger and action are JSON blobs for flexibility.
Rule Fire LogAudit row. Whenever a rule fires, log who/what was affected and what actions were taken. Enables the "rule audit log" view.
RecoveryA signed-negative LossEvent. The reverse motion. When a previously-lost device is recovered, the system creates a recovery_event that nets out the prior LossEvent's charges.
Naming convention to lock in early: the term "Loss" covers both outright loss and partial damage. The damage_components rows distinguish partial-damage incidents from total-loss incidents at the row level. Don't split into LossEvent vs. DamageEvent — same table, just different event_type values.
